## Lessons learned (for Claude to formalize on the [LL] issue — do NOT post yourself)
- **What happened:** Codex attacked multiple GitHub issues and left evidence in repo docs/tests/migrations, but did not initially leave progress comments on every issue.
- **Why it matters:** In the parallel-agent operating model, absent issue comments can make the team believe an issue is still untouched, causing duplicate investigation or duplicate implementation attempts.
- **Framework/skill/kit candidate:** Add a default "issue work audit" step to Codex handoff/closure: for every issue number touched, post a concise issue comment with scope, files/migrations, validation, and remaining ambiguity before final response.

- **What happened:** #639 initially proposed using `register_exclusion_asset` directly from `release-tag.yml`, but that RPC is intentionally declarant/self-service and depends on `auth.uid()` ownership.
- **Why it matters:** GitHub Actions runs with automation/service credentials; forcing it through a human-owned self-service RPC would either fail or encourage unsafe credential workarounds.
- **Framework/skill/kit candidate:** For automation writing into human-governed registries, add a narrow service-role RPC with explicit asset type, digest-only inputs, and no proof-byte forging; keep human self-service RPCs unchanged.

- **What happened:** #234 had MCP/OAuth refresh code in Worker SSR routes reading `import.meta.env.PUBLIC_SUPABASE_*`, while production deployment only guaranteed those values during `npm run build`.
- **Why it matters:** A missing runtime anon key makes server-side refresh return null without fetching Supabase, so the Claude connector falls back to the ~1h access-token lifetime even though `offline_access` and refresh-token support are advertised.
- **Framework/skill/kit candidate:** For Cloudflare Worker SSR OAuth/proxy paths, resolve operational credentials from `env` first and add a contract that `wrangler.toml` publishes any public runtime vars needed by deployed handlers; build-time env alone is not enough.

- **What happened:** During #678 QA, full Supabase anon JWT literals in `wrangler.toml` were caught as a release-blocking pre-commit/token-scanner risk.
- **Why it matters:** Supabase anon keys are public by design, but committing a full JWT literal in config collides with repo guardrails and can block or distract the release.
- **Framework/skill/kit candidate:** For public JWT-like config values, prefer runtime secrets/vars and add a contract asserting no full `eyJ...` token appears in deploy config or source.
